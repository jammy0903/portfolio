자 기 소 개 서

하드웨어의 언어를 사람의 언어로 바꾸는 개발자

저는 IoT 디바이스가 보내는 데이터를, 사람이 판단할 수 있는 화면으로 만드는 개발자입니다. 현재 아이앤씨테크놀로지에서 470대의 아크차단기가 보내는 전압·전류·아크 신호 데이터를 실시간으로 수집하고, 장애 발생 시 즉각 카카오톡 알림을 발송하는 IoT 모니터링 대시보드를 1인 개발·운영하고 있습니다.

입사 당시 이 시스템은 화면 디자인이 미완성이고 실시간 데이터 연동도 불안정한 프로토타입 단계에 머물러 있었습니다. 개발 문서도, 프론트엔드 시니어도 부재한 환경에서 Figma로 UI를 직접 재설계하고, 프론트엔드·백엔드·인프라 전체 스택을 단독으로 구축하여 6개월 만에 정식 B2B 서비스를 런칭했습니다. 영업팀으로부터 "모니터링 대시보드가 성남시 전통시장 수주의 80%를 결정했다"는 평가를 받았습니다.

구조를 이해해야 구조를 바꿀 수 있습니다

증상을 우회하는 대신 구조를 고치면, 한 번의 수정이 여러 문제를 동시에 해결합니다. 저는 문제가 생겼을 때 눈앞의 증상이 아니라, 그 아래의 구조를 먼저 봅니다.

초당 수백 건씩 쏟아지는 웹소켓 데이터로 프론트엔드 UI가 멈추는 문제가 발생했을 때, 레거시에서 사용 중이던 방식은 1초마다 서버에 변경 여부를 확인하는 폴링 기반이라 근본적으로 비효율적이었습니다. 대신 장애 데이터는 지연 0ms로 즉시 전송하되, 의미 없는 반복 데이터만 배치 처리하는 3단계 우선순위 매니저를 직접 설계하여 불필요한 렌더링을 80% 줄였습니다.

다중 게이트웨이 알림 수신 기능을 개발할 때도 마찬가지였습니다. 기존 DB에서 전화번호가 PK로 설정되어 있어, 동일 번호로 여러 게이트웨이의 알림을 받는 것 자체가 구조적으로 불가능한 상태였습니다. 테이블 추가나 API 전면 수정 같은 큰 공사 대신, PK를 UUID로 교체하고 phone+role에 UniqueConstraint를 적용하는 최소 변경 설계를 도출하여 DB·백엔드·프론트엔드 전 범위를 1일 내에 해결했습니다.

원리를 보는 눈이, 더 나은 코드를 만듭니다

저는 동작하는 코드보다 이해한 코드를 신뢰합니다. 그 믿음을 실천하기 위해 업무 외 시간에 CodeInsight라는 코드 실행 시각화 플랫폼을 만들고 있습니다.

대학에서 C를 배울 때, 포인터와 메모리 구조를 직관적으로 보여주는 도구가 없어 답답했던 경험에서 출발한 프로젝트입니다. C, Python, JavaScript, Java 4개 언어의 코드 실행 과정을 메모리 레벨에서 단계별로 시각화하는 인터랙티브 학습 플랫폼으로, 각 언어의 디버깅 메커니즘이 완전히 다름에도 하나의 통합 스키마로 정규화하는 아키텍처를 설계했습니다. TypeScript 517개 파일, 74,700줄, 커밋 386회의 개인 프로젝트이며 현재 배포·운영 중입니다.

보안 분야에서도 같은 태도로 임했습니다. KITRI 화이트햇스쿨 1기에서 메신저 포렌식 분석 도구를, KISIA S-개발자 2기에서 웹 취약점 CVE 매칭 프로그램을 개발했습니다. 두 과정 모두 학업과 병행하며 연구에 몰입했고, 이 경험은 업무 범위 밖의 문제도 주도적으로 해결하는 현재의 태도로 이어지고 있습니다.

혼자 해결하되, 혼자 일하지는 않습니다

기술적 역량만큼 중요한 것은 함께 일하는 방법을 아는 것이라고 생각합니다. 1인 개발 환경에서 일하고 있지만, 협업의 교훈은 오히려 더 선명하게 배웠습니다.

KISIA S-개발자 팀 프로젝트에서 문서 담당을 맡았을 때, 팀원들이 각자 작업만 하고 공유를 하지 않는 상황에서 "알아서 공유해주겠지"라고 기다렸다가, 팀원의 의도와 다른 방향으로 문서를 작성하는 실수를 했습니다. 그때 깨달은 것은, 소통의 책임은 기다리는 쪽이 아니라 먼저 물어보는 쪽에 있다는 것이었습니다.

이후 회사에서는 업무 완료 시 이메일로 진행 상황과 결과를 기록하는 보고 체계를 직접 만들었고, 이를 통해 비개발 직군인 영업팀과의 요구사항 소통도 원활해졌습니다. 도커 컨테이너가 회사 VPN IP를 점유하여 충돌이 발생했을 때에도, 담당 범위 밖의 인프라 문제였지만 직접 나서서 원인을 분석하고 해결한 뒤 팀에 공유했습니다. 내 담당이 아니더라도, 팀 전체의 흐름이 막히면 먼저 움직이는 것이 협업이라고 생각합니다.

다음 사람이 헤매지 않는 코드를 남기겠습니다

저는 문서 0%인 코드베이스를 물려받은 사람입니다. 그래서 IA, 시퀀스 다이어그램, 플로우차트를 직접 만들어 문서화 체계를 구축했습니다. "다음 사람은 나처럼 헤매지 않았으면 좋겠다"는 것이 그 이유였습니다.

궁극적으로 저는, 함께 일하는 동료가 저와 같은 여정을 겪을 때 혼자 헤매는 시간을 줄여줄 수 있는 개발자가 되고 싶습니다. 문서를 남기고, 아키텍처를 그려주고, 코드 리뷰로 함께 고민하는 것. 어떤 팀에서든 구조를 이해하고, 기록을 남기고, 먼저 소통하는 개발자로서 팀의 생산성에 기여하겠습니다.
