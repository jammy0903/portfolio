1. 소프트웨어 테스트 원칙 7가지 빈출

IEEE/ISTQB 기반 — 원칙 이름과 한 줄 정의 암기

원칙 7가지 — 순서대로 암기
① 결함 존재 증명

테스트는 결함이 있음을 증명할 뿐, 없음을 증명하지는 못한다.

② 완벽한 테스트 불가

모든 경우의 수를 테스트하는 것은 불가능. 위험 기반 우선순위 필요.

③ 조기 테스트

개발 초기부터 테스트 시작. 결함 발견이 늦을수록 수정 비용 기하급수적 증가.

④ 결함 집중 (Defect Clustering)

결함의 80%는 20% 모듈에 집중. 파레토 법칙 적용.

⑤ 살충제 역설 (Pesticide Paradox)

동일 테스트 케이스 반복 시 새 결함 발견 불가. 주기적으로 테스트 케이스를 갱신해야 함.

⑥ 테스트는 정황 의존

항공 소프트웨어와 쇼핑몰은 테스트 방법이 다름. 정황에 맞는 테스트 설계.

⑦ 오류 없음의 오류

결함이 없어도 사용자 요구에 맞지 않으면 품질이 좋은 게 아님.

2. 화이트박스 vs 블랙박스 테스트 빈출

소스 코드를 보느냐 안 보느냐로 구분

화이트박스 테스트 (White-box) — 구조 기반
소스 코드의 내부 구조·로직을 보면서 테스트. 개발자가 수행.
기법설명
기초 경로 검사사이클로매틱 복잡도 기반 독립 경로 도출
루프 검사단순·중첩·연결 루프 경계값 테스트
데이터 흐름 검사변수 정의-사용 구간 추적
조건 검사논리 조건 T/F 조합 테스트
→ 커버리지(구문·결정·조건 등) 기준으로 측정
블랙박스 테스트 (Black-box) — 명세 기반
소스 코드를 모르는 상태에서 기능 명세 기반으로 테스트. QA·사용자가 수행.

경계값 분석경계 근처 값(최솟값-1, 최솟값, 최솟값+1) 테스트
원인-효과 그래프입력(원인)↔출력(효과) 관계 그래프화
의사결정 테이블조건 조합표로 모든 경우 테스트
상태 전이 테스트시스템 상태 변화 경로 테스트
오류 예측경험 기반 오류 발생 가능성 높은 케이스
그레이박스 테스트: 화이트+블랙 혼합. 내부 일부 정보를 알고 테스트.  |  정적 테스트: 코드 실행 없이 검토(코드리뷰·인스펙션·워크스루).  |  동적 테스트: 코드 실행해서 확인.

3. 화이트박스 커버리지 종류 빈출

구문 → 결정 → 조건 → 조건/결정 → MC/DC → 다중조건 (강도 순)

커버리지 6종 — 약한 것에서 강한 순서
커버리지영문달성 조건예시
구문 커버리지 Statement Coverage 모든 문장(라인)을 최소 1번 실행 if문 안의 코드가 실행됐는지
결정 커버리지 Decision / Branch Coverage 각 분기(T/F)를 최소 1번씩 실행 if→true 경로, if→false 경로 모두
조건 커버리지 Condition Coverage 각 개별 조건이 T/F 모두 되도록 (A and B)에서 A·B 각각 T/F
조건/결정 커버리지 Condition/Decision 결정 + 조건 커버리지 동시 만족 분기 T/F + 개별 조건 T/F
변경 조건/결정 커버리지 MC/DC 각 조건이 독립적으로 결정에 영향을 줘야 함 항공·안전 소프트웨어 필수 기준
다중 조건 커버리지 Multiple Condition 조건의 모든 조합 실행 (가장 강함) 조건 n개 → 2ⁿ 케이스 필요
커버리지 예시 — if (A && B) 코드로 이해
코드

    // 블록 X
} else {
    // 블록 Y
}

커버리지필요 케이스
구문A=T,B=T (블록X) + A=F (블록Y)
결정결과=T / 결과=F 각 1회
조건A:T/F 각1, B:T/F 각1
MC/DCA만 변경 시 결과 변경 케이스 필요
다중조건TT, TF, FT, FF 4가지 모두

단계대상수행자내용
단위 테스트 개별 모듈·함수·클래스 개발자 최소 단위 기능 검증. 드라이버·스텁 사용. 화이트박스 위주.
통합 테스트 모듈 간 인터페이스 개발자/QA 모듈 연결 후 인터페이스·데이터 흐름 검증.
시스템 테스트 전체 통합 시스템 QA팀 기능·성능·보안·신뢰성 등 비기능 요구사항 검증.
인수 테스트 사용자 요구사항 사용자/고객 요구사항 충족 여부 최종 확인. 알파·베타 테스트 포함.
인수 테스트 세부 — 알파 vs 베타
알파 테스트 (Alpha Test)
  • 개발자 환경에서 사용자가 직접 테스트
  • 개발자가 옆에서 지켜봄
  • 출시 전 내부 테스트
베타 테스트 (Beta Test)
  • 실제 사용자 환경에서 테스트
  • 개발자 없이 사용자가 직접 사용
  • 출시 직전 외부 공개 테스트

5. 통합 테스트 방식 4종 빈출

하향식 · 상향식 · 샌드위치 · 빅뱅 — 스텁/드라이버 구분이 핵심

방식별 비교
방식방향필요한 것장점단점
하향식 (Top-down) 상위 → 하위 스텁 (Stub)
하위 모듈 가짜		 초기부터 주요 흐름 검증 가능 하위 모듈 오류 발견 늦음
상향식 (Bottom-up) 하위 → 상위 드라이버 (Driver)
상위 모듈 가짜		 하위 모듈 충분히 검증 전체 흐름 파악이 늦음
샌드위치 (Sandwich) 위+아래 동시 스텁 + 드라이버 모두 하향식+상향식 장점 결합 비용·관리 복잡
빅뱅 (Big-bang) 전체 한꺼번에 없음 (모든 모듈 완성) 단순, 추가 코드 불필요 결함 위치 파악 어려움. 소규모 적합.
스텁 vs 드라이버 — 헷갈림 방지
스텁 (Stub)

하위 모듈을 임시로 대체하는 가짜 모듈.
하향식 통합에서 사용.
실제 하위 모듈이 완성되기 전에 호출 받을 수 있게 최소 기능만 구현.

드라이버 (Driver)

상위 모듈을 임시로 대체하는 가짜 모듈.
상향식 통합에서 사용.
하위 모듈을 호출해주는 역할. 실제 상위 모듈 완성 전에 하위 모듈 테스트 가능.

⚡ 암기: 하향식=스텁(아래가 가짜), 상향식=드라이버(위가 가짜). "하스 상드" — 하향·스텁 / 상향·드라이버

6. 특수 테스트 유형

회귀·성능·보안·연기 등

자주 나오는 특수 테스트
회귀 테스트 (Regression)

수정·변경 후 기존 기능에 이상이 없는지 재테스트. 변경이 다른 부분을 망가뜨리지 않았는지 확인.

연기 테스트 (Smoke Test)

빌드 후 핵심 기능만 빠르게 돌려보는 테스트. 정상 동작 여부만 확인. 실패 시 더 이상 진행 안 함.

성능 테스트

응답 시간·처리량·자원 사용률 측정. 부하·스트레스·볼륨·내구성 테스트 포함.

부하 테스트 (Load)

정상 범위의 최대 부하에서 시스템 성능 측정.

스트레스 테스트

정상 범위를 초과한 극한 상황에서 한계점 파악.

탐색적 테스트

테스트 케이스 없이 실시간으로 탐색·설계·실행을 동시에. 경험 기반.

7. 사이클로매틱 복잡도 빈출

V(G) = E - N + 2 공식 + 예시

공식 + 계산법
공식 2가지
V(G) = E − N + 2
  • E = 간선(Edge, 화살표) 수
  • N = 노드(Node, 동그라미) 수
  • V(G) = 결정(Decision) 수 + 1
    (if·for·while·case 각각 +1)
예시 계산 
int func(int n) {
    int sum = 0;          // 결정 없음
    for(int i=0; i<n; i++) { // 결정 1
        if(i % 2 == 0)    // 결정 2
            sum += i;
    }
    return sum;
}

결정 수 = 2개 (for, if)
V(G) = 2 + 1 = 3
복잡도 기준: 1~10 단순 / 11~20 적당히 복잡 / 21~50 매우 복잡 / 50 초과 테스트 불가 수준. 복잡도 높을수록 테스트 케이스 더 필요.

8. 결함·테스트 관련 용어 빈출

에러·결함·장애 구분 + 테스트 산출물

에러 vs 결함 vs 장애 — 3단계 구분
에러 (Error)
개발자가 코딩 중 범한 실수. 소스코드의 잘못.
→
결함 (Defect / Bug / Fault)
에러로 인해 소프트웨어에 내재된 잘못. 잠재적 오류.
→
장애 (Failure)
결함이 실행되어 실제로 나타난 오동작. 사용자가 경험.
테스트 산출물 · 용어
테스트 케이스

특정 기능 검증을 위한 입력값·실행 조건·기대 결과 명세서.

테스트 시나리오

테스트 케이스들의 실행 순서·흐름을 정의한 문서.

테스트 오라클

테스트 결과가 올바른지 판단하는 기준. 기대 출력값.

목 (Mock) 객체

실제 객체를 모방한 가짜 객체. 단위 테스트에서 의존성 대체.

테스트 하네스

테스트 환경 구축을 위한 드라이버·스텁 등 도구 모음.

결함 밀도

소스코드 규모 대비 발견된 결함 수. 결함 수 ÷ 코드 라인 수.

정적 분석 기법 — 실행 없이 결함 찾기
코드 리뷰 (Code Review)

개발자 간 소스코드 상호 검토. 비형식적.

인스펙션 (Inspection)

체크리스트 기반 공식 검토 회의. 사회자·작성자·검토자 역할 분리.

워크스루 (Walkthrough)

작성자가 직접 설명하며 팀원이 비공식 검토. 인스펙션보다 가벼움.

에러→결함→장애 흐름: 개발자 실수(에러) → 코드에 버그(결함) → 실행 시 오작동(장애).  |  테스트 원칙 ⑤ 살충제 역설: 같은 케이스 반복은 새 결함 발견 못함 → 주기적 갱신 필요.
테스트는 화이트박스·블랙박스 기법 이름과 단위→통합→시스템→인수 순서, 하향식=스텁 / 상향식=드라이버, 사이클로매틱 V(G)=E-N+2 공식이 핵심 빈출입니다.
