디자인 패턴 상세 정리

디자인 패턴은 이름만 외우면 금방 섞입니다. 그래서 이번 페이지는 패턴 하나마다 “무엇을 해결하는지”, “언제 쓰는지”, “비슷한 패턴과 뭐가 다른지”, “시험에서는 어떤 문장으로 나오기 쉬운지”를 같이 붙였습니다. 짧은 암기표가 아니라, 패턴을 헷갈리지 않게 잡아 주는 레퍼런스형 정리입니다.

생성 패턴 구조 패턴 행위 패턴 기출 + 미출제 핵심

생성 패턴

객체를 어떤 방식으로 만들지에 초점을 둡니다. 시험에서는 “하나만 만든다”, “관련 객체군을 묶어 만든다”, “단계적으로 만든다”, “복제한다” 같은 표현으로 나옵니다.

구조 패턴

클래스와 객체를 어떻게 연결하고 감쌀지에 초점을 둡니다. “호환되지 않는 인터페이스를 맞춘다”, “기능을 덧붙인다”, “복잡한 하위 시스템을 감싼다” 같은 표현이 핵심입니다.

행위 패턴

객체끼리 어떻게 책임을 나누고 상호작용할지에 초점을 둡니다. “상태 변화를 알린다”, “알고리즘을 바꿔 낀다”, “순차 탐색한다”, “요청을 객체화한다” 같은 문장이 힌트입니다.

Observer Proxy Factory Method Singleton Visitor Abstract Factory Iterator Adapter Bridge Command Chain of Responsibility Strategy Decorator Composite Facade Builder Prototype

행위 패턴

Observer

한 객체의 상태가 바뀌었을 때, 그 객체에 의존하는 다른 객체들에게 자동으로 알려 주는 패턴입니다. 핵심은 “구독 관계”입니다. 누군가의 변화가 여러 곳에 동시에 반영되어야 할 때 씁니다.

출제: 2020-2회 10번, 2022-3회 3번 행위 패턴 핵심: 상태 변화 자동 통지

언제 쓰나

  • 한 데이터가 바뀌면 여러 화면, 모듈, 객체가 동시에 반응해야 할 때
  • 주체와 관찰자를 느슨하게 연결하고 싶을 때
  • 이벤트 기반 구조를 만들고 싶을 때

쉽게 이해하기

유튜브 채널을 구독해 두면 새 영상이 올라왔을 때 구독자들에게 알림이 갑니다. 채널이 주체(Subject), 구독자가 관찰자(Observer)입니다. 주체가 상태를 바꾸면 관찰자들이 자동 반응합니다.

헷갈리는 패턴

Strategy는 알고리즘을 갈아끼우는 패턴이고, Observer는 변화 사실을 여러 객체에 알리는 패턴입니다. 즉 Strategy는 “교체”, Observer는 “통지”가 핵심입니다.

시험 포인트

문제에서 “상태 변화”, “자동 통지”, “관련 객체들에 알림”, “의존 객체들이 자동 반응” 같은 문장이 나오면 Observer를 먼저 떠올리면 됩니다.

구조 패턴

Proxy

실제 객체 앞에 대리 객체를 하나 두고, 그 대리 객체가 접근을 대신 제어하는 패턴입니다. 핵심은 “대신 서서 통제한다”입니다.

출제: 2023-1회 11번, 2025-2회 7번 구조 패턴 핵심: 대리 객체

언제 쓰나

  • 실제 객체 접근 전에 권한 검사, 로깅, 캐시를 넣고 싶을 때
  • 생성 비용이 큰 객체를 바로 만들고 싶지 않을 때
  • 원격 객체나 무거운 객체를 대신 다루고 싶을 때

쉽게 이해하기

건물 출입구의 경비실이 Proxy에 가깝습니다. 실제 건물 내부로 바로 들어가는 게 아니라 경비실이 먼저 신원을 확인하고 접근을 허용합니다. 실제 대상 앞에 대리인이 있는 구조입니다.

헷갈리는 패턴

Decorator는 기능을 더 붙이는 데 초점이 있고, Proxy는 접근 통제와 대리 처리에 초점이 있습니다. 둘 다 “감싼다”는 점은 비슷하지만 목적이 다릅니다.

시험 포인트

“실제 객체 대신”, “대리 객체”, “접근 제어”, “권한 검사”, “가상 대리”라는 표현이 보이면 Proxy 쪽입니다.

생성 패턴

Factory Method

객체를 생성하는 책임을 상위 클래스가 직접 확정하지 않고, 하위 클래스가 어떤 객체를 만들지 결정하게 위임하는 패턴입니다. 핵심은 “생성 결정을 서브클래스에 넘긴다”입니다.

출제: 2021-3회 16번 생성 패턴 핵심: 생성 위임

언제 쓰나

  • 상위 로직은 유지하되 생성될 구체 객체만 바뀔 수 있을 때
  • 객체 생성 코드를 캡슐화해 확장에 유리하게 만들고 싶을 때
  • new를 직접 남발하지 않고 생성 책임을 분리하고 싶을 때

쉽게 이해하기

상위 공장이 “제품을 만들어라”라는 틀만 제공하고, 실제 어떤 브랜드 제품을 만들지는 각 하위 공장이 정하는 구조입니다. 공장 메서드는 같지만 만들어지는 제품은 다를 수 있습니다.

헷갈리는 패턴

Abstract Factory는 “관련된 객체군 전체”를 만드는 데 초점이 있고, Factory Method는 “한 종류의 객체 생성 결정”을 하위 클래스에 위임하는 데 초점이 있습니다.

시험 포인트

문제에서 “객체 생성을 서브클래스가 결정”, “생성 책임 위임”, “상위 클래스는 인터페이스만 제공” 같은 표현이 보이면 Factory Method입니다.

생성 패턴

Singleton

어떤 클래스의 인스턴스가 시스템 전체에서 하나만 존재하도록 보장하고, 그 인스턴스를 전역적으로 공유하게 만드는 패턴입니다.

출제: 2023-2회 11번 생성 패턴 핵심: 인스턴스 하나

언제 쓰나

  • 설정 관리자, 로그 관리자처럼 하나만 있어야 하는 객체일 때
  • 전역 접근이 필요하지만 객체를 여러 개 만들면 안 될 때

쉽게 이해하기

한 건물의 중앙 관리실이 하나만 있는 것과 비슷합니다. 어디서 접근하든 같은 관리실을 사용해야 혼선이 없습니다.

주의점

시험에서는 보통 장점만 묻지만, 실무에서는 전역 상태가 많아져 테스트가 어려워질 수 있습니다. 그래도 실기에서는 “인스턴스 하나”가 핵심 기억점입니다.

시험 포인트

“인스턴스를 하나만”, “전역 공유”, “단 하나의 객체”라는 문장이 나오면 거의 Singleton입니다.

행위 패턴

Visitor

자료 구조는 그대로 두고, 그 구조에 대해 수행할 연산만 별도의 객체로 분리하는 패턴입니다. 핵심은 “구조는 유지, 연산은 분리”입니다.

출제: 2023-2회 11번 행위 패턴 핵심: 연산 분리

언제 쓰나

  • 객체 구조는 자주 안 바뀌는데, 그 위에 새로운 연산은 자주 추가될 때
  • 객체마다 연산 로직을 흩뿌리지 않고 방문 객체에 모으고 싶을 때

쉽게 이해하기

박물관 전시품은 그대로 있는데, 해설사 역할만 바뀌는 것과 비슷합니다. 작품 구조는 안 바꾸고, 설명 방식만 따로 붙입니다.

헷갈리는 패턴

Iterator는 구조를 순회하는 데 초점이 있고, Visitor는 구조를 방문하면서 별도 연산을 적용하는 데 초점이 있습니다.

시험 포인트

“자료 구조 유지”, “연산만 분리”, “새로운 기능을 별도 객체로 추가” 같은 문장이 나오면 Visitor를 생각하면 됩니다.

생성 패턴

Abstract Factory

서로 관련 있는 객체들을 한 묶음으로 생성하는 패턴입니다. 핵심은 “관련 객체군을 통째로 만든다”입니다.

출제: 2024-1회 20번 생성 패턴 핵심: 객체군 생성

언제 쓰나

  • 서로 어울리는 객체 세트를 함께 만들어야 할 때
  • 운영체제별 UI처럼 제품군 단위 일관성이 중요할 때

쉽게 이해하기

북유럽 스타일 가구 세트, 클래식 스타일 가구 세트를 따로 파는 브랜드를 생각하면 됩니다. 의자, 책상, 조명이 서로 어울리는 세트로 나옵니다.

헷갈리는 패턴

Factory Method가 개별 제품 생성 위임이라면, Abstract Factory는 제품군 전체를 맞춰서 생성한다는 차이가 있습니다.

시험 포인트

“관련 있는 객체군”, “일관성 있는 제품군”, “묶어서 생성”이라는 말이 나오면 Abstract Factory 쪽입니다.

행위 패턴

Iterator

컬렉션의 내부 구조를 드러내지 않고, 원소를 순서대로 탐색할 수 있게 만드는 패턴입니다. 핵심은 “내부를 숨긴 채 순차 탐색”입니다.

출제: 2024-2회 10번 행위 패턴 핵심: 순차 탐색

언제 쓰나

  • 리스트, 트리 등 내부 구조는 다르지만 공통 방식으로 탐색하고 싶을 때
  • 컬렉션 구현 세부를 외부에 노출하고 싶지 않을 때

쉽게 이해하기

책장을 넘기면서 한 페이지씩 읽는 방식과 비슷합니다. 책 내부 제본 구조를 몰라도 페이지를 순서대로 볼 수 있습니다.

헷갈리는 패턴

Visitor는 방문하면서 연산을 적용하는 데 초점이 있고, Iterator는 어디까지나 “순회 방식 제공”이 핵심입니다.

시험 포인트

“내부 구조를 드러내지 않고”, “원소를 순차 탐색”, “반복자”라는 표현이 보이면 Iterator입니다.

구조 패턴

Adapter

호환되지 않는 두 인터페이스를 중간에서 맞춰 연결하는 패턴입니다. 핵심은 “안 맞는 규격을 변환해 붙인다”입니다.

출제: 2025-1회 14번 구조 패턴 핵심: 인터페이스 변환

언제 쓰나

  • 기존 코드를 바꾸지 않고 새 시스템과 연결하고 싶을 때
  • 인터페이스가 달라서 직접 연결이 안 될 때

쉽게 이해하기

해외 멀티탭에 국내 플러그를 꽂기 위한 변환 어댑터와 같습니다. 기기 자체를 바꾸지 않고, 연결 규격만 맞춥니다.

헷갈리는 패턴

Bridge는 추상화와 구현을 분리하는 구조이고, Adapter는 이미 존재하는 서로 다른 인터페이스를 이어 붙이는 구조입니다.

시험 포인트

“호환되지 않는 인터페이스”, “연결을 위해 변환”, “호환성 맞춤”이라는 문장이 나오면 Adapter입니다.

구조 패턴

Bridge

추상화와 구현을 분리해서 서로 독립적으로 확장할 수 있게 만드는 패턴입니다. 핵심은 “기능 계층과 구현 계층을 분리한다”입니다.

출제: 2022-3회 3번 구조 패턴 핵심: 추상화 / 구현 분리

언제 쓰나

  • 기능 종류와 구현 종류가 서로 따로 늘어날 수 있을 때
  • 상속만으로 조합 수가 폭발하는 것을 막고 싶을 때

쉽게 이해하기

리모컨 종류(기본형, 고급형)와 TV 브랜드(삼성, LG)가 따로 늘어나는 상황을 생각하면 됩니다. 리모컨과 TV를 분리하면 조합이 유연해집니다.

헷갈리는 패턴

Adapter는 이미 안 맞는 것을 연결하는 데 초점이 있고, Bridge는 애초에 구조를 분리해서 독립 확장하게 하는 데 초점이 있습니다.

시험 포인트

“추상화와 구현 분리”, “독립적 확장”, “상속 폭발 방지” 같은 표현이 나오면 Bridge를 생각하세요.

행위 패턴

Command

요청 자체를 객체로 캡슐화하는 패턴입니다. 핵심은 “행동을 데이터처럼 다룬다”입니다. 그래서 실행, 저장, 취소, 큐잉 같은 처리가 쉬워집니다.

약출제 행위 패턴 핵심: 요청 객체화

언제 쓰나

  • 버튼을 눌렀을 때 실행할 동작을 유연하게 바꾸고 싶을 때
  • 실행 취소, 재실행, 작업 기록 기능이 필요할 때

쉽게 이해하기

리모컨 버튼 하나가 곧 “명령 객체”라고 생각하면 됩니다. 버튼은 요청을 담고 있고, 실제 TV가 그 요청을 수행합니다.

헷갈리는 패턴

Strategy는 알고리즘 교체, Command는 요청 자체를 객체로 다루는 패턴입니다. 하나는 방법, 하나는 요청입니다.

시험 포인트

“요청을 객체로 캡슐화”, “실행 취소”, “재실행”, “큐에 저장” 같은 키워드가 보이면 Command입니다.

행위 패턴

Chain of Responsibility

요청을 처리할 수 있는 객체들을 사슬처럼 연결해 두고, 각 객체가 처리할 수 있으면 처리하고 아니면 다음 객체로 넘기는 패턴입니다.

약출제 행위 패턴 핵심: 순차 전달

언제 쓰나

  • 누가 처리할지 미리 고정하지 않고, 가능한 처리자에게 순서대로 맡기고 싶을 때
  • 결재 라인, 예외 처리 체인처럼 단계별 넘김 구조가 필요할 때

쉽게 이해하기

회사 결재선에서 팀장 → 부장 → 이사 순으로 문서를 올리는 구조와 비슷합니다. 앞사람이 처리 못 하면 뒤로 넘어갑니다.

헷갈리는 패턴

Command는 요청을 객체화하고, Chain of Responsibility는 그 요청을 누가 처리할지 체인으로 넘기는 방식입니다.

시험 포인트

“사슬처럼 연결”, “순차 처리”, “처리 못 하면 다음 객체로 전달”이라는 문장이 보이면 이 패턴입니다.

행위 패턴

Strategy

관련된 알고리즘들을 각각 캡슐화하고, 필요에 따라 서로 바꿔 끼울 수 있게 만드는 패턴입니다. 핵심은 “알고리즘 교체”입니다.

미출제 최우선 행위 패턴 핵심: 알고리즘 교체

언제 쓰나

  • 정렬 방식, 할인 방식, 경로 탐색 방식처럼 방법만 바뀔 수 있을 때
  • if-else로 알고리즘 분기가 계속 늘어나는 것을 줄이고 싶을 때

쉽게 이해하기

길찾기 앱에서 자동차 경로, 도보 경로, 대중교통 경로를 선택하는 것과 같습니다. 목적은 같지만 알고리즘만 바뀝니다.

헷갈리는 패턴

State와 이름이 헷갈릴 수 있지만, Strategy는 사용자가 혹은 문맥이 “방법을 선택”하는 쪽이고, State는 객체 상태 변화에 따라 동작이 달라지는 쪽입니다.

시험 포인트

“알고리즘군”, “교체 가능”, “캡슐화”, “상황에 따라 다른 방식 선택”이라는 문장이 나오면 Strategy입니다.

이 패턴은 원본에서도 `최우선 쪽집게`로 찍혀 있습니다. 미출제지만 카테고리 특성상 다음 타자로 묻기 좋습니다.

구조 패턴

Decorator

객체를 감싸면서 기능을 동적으로 덧붙이는 패턴입니다. 핵심은 “상속 대신 포장해서 기능 추가”입니다.

미출제 최우선 구조 패턴 핵심: 기능 덧붙이기

언제 쓰나

  • 기존 객체 코드를 바꾸지 않고 기능을 계속 추가하고 싶을 때
  • 상속 조합이 너무 많아지는 것을 피하고 싶을 때

쉽게 이해하기

기본 커피에 샷 추가, 시럽 추가, 휘핑 추가를 차례대로 붙이는 것과 비슷합니다. 기본 객체를 감싸면서 기능을 더합니다.

헷갈리는 패턴

Proxy도 감싸지만, Proxy는 대리와 통제, Decorator는 기능 추가가 목적입니다. 둘 다 래핑하지만 의도가 다릅니다.

시험 포인트

“동적으로 기능을 덧붙인다”, “객체를 감싼다”, “상속 대신 기능 확장” 같은 문장이 나오면 Decorator입니다.

실기에서 구조 패턴을 묻는데 “기능을 덧붙인다”가 보이면 거의 Decorator입니다. Proxy와 구별만 정확히 하면 됩니다.

구조 패턴

Composite

부분과 전체를 동일한 방식으로 다룰 수 있게 만드는 패턴입니다. 핵심은 “부분-전체 계층을 하나의 인터페이스로 본다”입니다.

미출제 구조 패턴 핵심: 부분 / 전체 동일 취급

언제 쓰나

  • 파일 / 폴더, 부품 / 조립품처럼 계층 구조를 통일된 방식으로 다뤄야 할 때
  • 클라이언트가 개별 객체와 묶음 객체를 구분하지 않고 처리하게 하고 싶을 때

쉽게 이해하기

파일 시스템에서 파일과 폴더를 떠올리면 쉽습니다. 폴더 안에는 파일도 있고 하위 폴더도 있습니다. 그런데 사용자는 둘 다 목록에서 거의 같은 방식으로 다룹니다.

헷갈리는 패턴

Decorator는 감싸서 기능을 추가하고, Composite는 계층 구조를 통합 인터페이스로 다루는 점이 다릅니다.

시험 포인트

“부분-전체 계층”, “동일하게 다룬다”, “트리 구조”, “계층 구조”라는 말이 보이면 Composite입니다.

구조 패턴

Facade

복잡한 하위 시스템을 직접 다루지 않도록, 그 앞에 단순한 인터페이스를 하나 제공하는 패턴입니다. 핵심은 “복잡함을 단순 창구 하나로 감춘다”입니다.

미출제 최우선 구조 패턴 핵심: 단순 창구

언제 쓰나

  • 하위 시스템이 너무 복잡해 직접 호출하기 어렵고, 사용법을 단순화하고 싶을 때
  • 클라이언트가 내부 세부를 몰라도 되게 만들고 싶을 때

쉽게 이해하기

은행 창구 직원이 여러 내부 부서를 대신 연결해 주는 구조와 비슷합니다. 사용자는 창구 하나만 보면 되고, 내부 처리는 창구 뒤에서 복잡하게 돌아갑니다.

헷갈리는 패턴

Adapter는 인터페이스 불일치를 맞추는 데 초점이 있고, Facade는 복잡한 시스템을 쉽게 쓰게 해 주는 데 초점이 있습니다.

시험 포인트

“복잡한 하위 시스템”, “단순 인터페이스”, “통합 창구”, “사용을 쉽게”라는 문장이 나오면 Facade입니다.

Facade는 “안쪽은 복잡한데 바깥은 단순하게 한 번에 쓰게 한다”는 문장을 기억하면 거의 틀리지 않습니다.

생성 패턴

Builder

복잡한 객체를 한 번에 만들지 않고, 단계적으로 조립하면서 생성하는 패턴입니다. 핵심은 “복잡한 생성을 단계로 분해한다”입니다.

미출제 최우선 생성 패턴 핵심: 단계적 생성

언제 쓰나

  • 생성자 인자가 너무 많고 조합이 복잡할 때
  • 만드는 순서와 구성 요소를 분리하고 싶을 때

쉽게 이해하기

햄버거를 만들 때 빵, 패티, 채소, 소스를 순서대로 조립하는 것과 같습니다. 한 번에 만드는 게 아니라 단계별로 쌓아 갑니다.

헷갈리는 패턴

Abstract Factory는 관련 객체군을 묶어서 만들고, Builder는 하나의 복잡한 객체를 단계적으로 만들어 갑니다.

시험 포인트

“단계적으로 생성”, “복잡한 객체 조립”, “생성 과정 분리”라는 문장이 보이면 Builder입니다.

Builder는 최근 원본에서 강하게 찍힌 미출제 핵심입니다. “복잡한 객체를 단계적으로 만든다”는 표현을 통째로 외우는 게 좋습니다.

생성 패턴

Prototype

기존 객체를 복제해서 새로운 객체를 만드는 패턴입니다. 핵심은 “새로 만들지 않고 복사해 만든다”입니다.

미출제 최우선 생성 패턴 핵심: 객체 복제

언제 쓰나

  • 객체 생성 비용이 크고, 기존 객체와 비슷한 객체를 많이 만들어야 할 때
  • 생성 로직보다 복제가 더 간단할 때

쉽게 이해하기

도장 찍듯이 원본 하나를 만들어 두고 사본을 계속 찍어 내는 구조입니다. 템플릿 문서를 복사해서 새 문서를 만드는 느낌으로 이해하면 쉽습니다.

헷갈리는 패턴

Builder는 조립해서 만들고, Prototype은 복제해서 만듭니다. 시험에서는 “단계적”인지 “복제”인지가 갈림길입니다.

시험 포인트

“기존 객체 복제”, “원형 객체”, “사본 생성” 같은 표현이 보이면 Prototype입니다.

시험에서는 보통 이름 + 분류(생성 / 구조 / 행위) + 한 줄 뜻 조합으로 묻습니다. 하지만 진짜 잘 외우려면 “무엇을 해결하는 패턴인지”까지 같이 잡아야 헷갈리지 않습니다. 특히 Strategy / Decorator / Facade / Builder / Prototype는 미출제 핵심으로 계속 눈에 익혀 두는 편이 좋습니다.
