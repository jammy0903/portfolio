개발 방법론 요구사항 분석 아키텍처 패턴 SOLID 결합도 응집도 UML 모듈·인터페이스 디자인 패턴

1. 소프트웨어 개발 방법론

전통적 계획 중심 vs 애자일 — 구조·구성원·산출물 차이 암기

전통적 / 계획 중심 방법론
폭포수 (Waterfall)
  • 요구분석→설계→구현→테스트→유지보수
  • 단계별 순서 진행, 이전 단계 복귀 어려움
  • 문서화·계획 중심, 안정적 요구사항에 적합
  • 장: 관리 쉬움 / 단: 변경 대응 어려움
나선형 (Spiral) — 보엠(Boehm)
  • 4단계 반복: 계획 → 위험 분석 → 개발·검증 → 고객 평가
  • 위험 분석이 핵심 — 대규모·고위험 프로젝트
  • 점진적·반복적 개발
프로토타이핑 (Prototyping)
  • 요구사항 불명확할 때 시제품 먼저 제작
  • 사용자 피드백 반영 후 최종 개발
  • 진화적 프로토타입 vs 폐기형 프로토타입
구조적 방법론
  • 프로세스(기능) 중심 분석
  • DFD(데이터 흐름도)·DD·미니 명세서 사용
  • 하향식(Top-Down) 분할 설계
정보공학 방법론
  • 데이터 중심 접근
  • 4단계: 정보전략계획→업무영역분석→시스템설계→구축
  • ERD 활용
CBD (컴포넌트 기반)
  • 재사용 가능한 컴포넌트 조립으로 시스템 구축
  • 인터페이스 기반 설계
  • 개발 기간 단축, 유지보수 용이
애자일 방법론 (Agile) — 짧은 반복, 변화 수용, 작동 소프트웨어 우선
Scrum 최빈출
  • 스프린트: 2~4주 고정 반복 주기
  • 역할 3가지:
    • 스크럼 마스터 — 방해 요소 제거, 프로세스 관리
    • 프로덕트 오너 — 제품 백로그 우선순위 결정
    • 개발팀 — 자기 조직화, 실제 구현
  • 산출물: 제품 백로그 → 스프린트 백로그 → 실행 소프트웨어
  • 이벤트: 스프린트 계획 / 데일리 스크럼(15분, 3질문) / 스프린트 리뷰 / 회고
  • 번다운 차트: 남은 작업량 시각화
XP (익스트림 프로그래밍)
  • 5가지 가치: 의사소통·단순성·피드백·용기·존중
  • TDD: 테스트 먼저 → 코드 구현
  • 페어 프로그래밍: 2인 1조
  • 지속적 통합(CI), 리팩토링
  • 소규모 릴리즈, 집단 코드 소유권
  • 온사이트 고객 (현장 상주)
Kanban / Lean
  • Kanban: 할 일→진행 중→완료 보드
  • WIP(진행 중 작업) 제한 — 병목 제거
  • 스프린트 없음, 연속적 흐름
  • Lean: 낭비 7종 제거
  • 가치 흐름 최적화, 빠른 피드백
애자일 선언 4가치: 프로세스·도구 < 개인·상호작용 / 포괄적 문서 < 작동하는 소프트웨어 / 계약협상 < 고객협력 / 계획따르기 < 변화대응  |  폭포수 vs 애자일: 폭포수=계획·문서·순차 / 애자일=반복·협업·변화수용

2. 요구사항 분석

기능적 vs 비기능적 / 도출·명세·검증 기법

요구사항 분류
기능적 요구사항 (Functional)
  • 시스템이 무엇을 해야 하는가
  • 기능·서비스·동작 정의
  • 예: "로그인 기능을 제공해야 한다"
비기능적 요구사항 (Non-Functional)
  • 시스템이 얼마나 잘 해야 하는가
  • 성능·신뢰성·보안·확장성·이식성·유지보수성
  • 예: "응답 시간 2초 이하", "99.9% 가용성"
요구사항 도출 기법
인터뷰·설문

이해관계자 직접 질의, 정량 데이터 수집

브레인스토밍·JAD 워크숍

그룹 창의적 아이디어, 공동 요구사항 정의

시나리오·유스케이스

사용 시나리오 기반, 사용자 관점 요구사항

프로토타이핑

시제품으로 요구사항 명확화, 피드백 반영

관찰 (민족지학)

실제 업무 현장 관찰로 묵시적 요구 파악

문서 분석

기존 시스템·업무 문서 검토

요구사항 검증 기법 (검토 방법)
동료 검토 (Peer Review)

작성자 포함 팀원이 문서 검토. 비공식.

워크스루 (Walk-through)

작성자가 직접 설명, 팀원이 질의·검토. 비공식.

인스펙션 (Inspection) 중요

역할 지정(진행자·작성자·검토자·기록자). 가장 공식적·효과적.

럼바우(Rumbaugh) OMT 분석: ① 객체 모델링(Object) — 클래스·속성·관계, ERD 유사   ② 동적 모델링(Dynamic) — 상태 변화, 상태 다이어그램   ③ 기능 모델링(Function) — 데이터 흐름, DFD

3. 아키텍처 패턴

시스템 전체 구조 설계 방식 — MVC·Layered 빈출

기본 구조 패턴
계층형 (Layered)
  • 상위→하위 단방향 의존, 계층 간 분리
  • 표현층·비즈니스층·데이터층
  • OSI 7계층 같은 구조
클라이언트-서버
  • 서비스 제공자(서버) / 요청자(클라이언트) 분리
  • 서버는 다수 클라이언트 동시 처리
파이프-필터
  • 데이터가 파이프 통해 필터 단계 순서 통과
  • Unix 파이프: ls | grep | wc
  • 컴파일러, ETL 파이프라인
이벤트 기반 (Event-Driven)
  • 이벤트 발생 시 핸들러 실행
  • 발행-구독(Pub-Sub) 패턴
  • Kafka, RabbitMQ
마스터-슬레이브
  • 마스터가 슬레이브에 작업 분산·통합
  • 병렬 처리, 장애 허용에 활용
브로커 (Broker)
  • 컴포넌트 간 중재자 역할
  • CORBA, 원격 서비스 호출
MVC 계열 — UI 분리 패턴 (빈출)
MVC (Model-View-Controller) 빈출
  • Model: 데이터 + 비즈니스 로직
  • View: 화면 출력 (UI)
  • Controller: 사용자 입력 처리, Model↔View 연결
  • View가 Model에 직접 의존 → 복잡해질 수 있음
MVP (Model-View-Presenter)
  • View와 Model 완전 분리
  • Presenter가 모든 로직 담당
  • View는 UI만, 테스트 용이
MVVM (Model-View-ViewModel)
  • ViewModel이 View 상태를 Observable로 제공
  • 데이터 바인딩으로 자동 동기화
  • Angular, Vue, WPF에서 사용
현대 분산 아키텍처
MSA (마이크로서비스)
  • 서비스별 독립 배포·확장·기술 스택
  • API Gateway로 단일 진입점
  • REST·메시지 큐로 서비스 간 통신
SOA (서비스 지향 아키텍처)
  • 비즈니스 기능을 서비스 단위로 분리
  • ESB(엔터프라이즈 서비스 버스)로 통합
  • MSA보다 큰 서비스 단위
EDA (이벤트 주도 아키텍처)
  • 이벤트 발생·감지·반응 기반 설계
  • 비동기·느슨한 결합, 확장성 우수

4. SOLID 원칙 (객체지향 설계 5원칙)

좋은 OOP 설계 기준 — 영문 약자·한국어 이름·핵심 한 줄 암기

단일 책임

클래스는 하나의 책임만. 변경 이유가 하나여야 함.

확장에는 열려 있고, 수정에는 닫혀 있어야. 추상화·다형성 활용.

ISP

DRY: Don't Repeat Yourself — 중복 배제  |  KISS: Keep It Simple — 단순하게  |  YAGNI: You Aren't Gonna Need It — 필요한 것만 구현  |  응집도↑ + 결합도↓ = 좋은 모듈 설계

① 내용 결합 (Content)

한 모듈이 다른 모듈 내부 데이터·로직을 직접 참조·수정. 가장 위험.

② 공통 결합 (Common)

두 모듈이 전역 변수(공유 데이터)를 함께 사용. 전역변수 문제.

③ 외부 결합 (External)

외부 환경(파일·OS·하드웨어·표준 프로토콜)에 종속.

④ 제어 결합 (Control)

제어 신호(플래그·스위치)를 파라미터로 전달해 흐름 제어.

⑤ 스탬프 결합 (Stamp)

레코드·구조체 전체를 파라미터로 전달. 불필요 데이터 포함.

⑥ 자료 결합 (Data)

단순 스칼라 값만 파라미터로 전달. 가장 이상적.

① 기능적

② 순차적

앞 처리 출력 = 다음 처리 입력. 순서 관계 있음.

③ 통신적

같은 데이터를 사용하는 기능들의 묶음.

④ 절차적

관련 기능을 순서대로 수행. 자료 공유 없음.

⑤ 시간적

같은 시간에 실행되는 기능들. 초기화·종료 루틴.

⑥ 논리적

유사 기능 묶어두고 선택 플래그로 실행.

⑦ 우연적

아무 관련 없는 기능이 우연히 묶임. 최악.

7. UML 다이어그램

구조(정적) vs 행위(동적) 분류 + 클래스 다이어그램 관계 표기법

다이어그램 분류
구조 다이어그램 (정적 — 시스템 구조)
  • 클래스 다이어그램 — 클래스·속성·메서드·관계 빈출
  • 객체 다이어그램 — 특정 시점 인스턴스 상태
  • 컴포넌트 다이어그램 — 컴포넌트 구조·인터페이스
  • 배포 다이어그램 — 하드웨어 노드 배치
  • 패키지 다이어그램 — 패키지 간 의존 관계
행위 다이어그램 (동적 — 시스템 행위)
  • 유스케이스 다이어그램 — 사용자·시스템 상호작용 빈출
  • 시퀀스 다이어그램 — 시간 순 메시지 흐름 빈출
  • 활동 다이어그램 — 작업 흐름, 조건 분기
  • 상태 다이어그램 — 객체 상태 전이
  • 협업 다이어그램 — 객체 협력 관계
클래스 다이어그램 관계 — 화살표 방향+이름 암기 (빈출)
관계표기의미예시
연관 (Association)실선 (―→)클래스 간 관계가 있음. 양방향 or 단방향.학생 ― 수업
의존 (Dependency)점선 화살표 (···→)한 클래스가 다른 클래스를 일시적으로 사용메서드 파라미터로 사용
일반화/상속 (Generalization)실선 + 빈 삼각형 (△)상속 관계. 자식→부모 방향.개 → 동물
실체화 (Realization)점선 + 빈 삼각형인터페이스 구현Dog implements Animal
집합 (Aggregation)실선 + 빈 마름모 (◇)전체-부분. 부분은 독립 존재 가능.팀 ◇― 선수
합성 (Composition)실선 + 채워진 마름모 (◆)강한 전체-부분. 전체 소멸 시 부분도 소멸.집 ◆― 방
유스케이스 다이어그램 핵심
액터 (Actor)

시스템 외부에서 상호작용하는 존재. 사람 또는 외부 시스템. 막대 인간 기호.

포함 관계 «include»

반드시 포함 실행되는 공통 기능. 화살표: 기본 UC → 포함 UC

확장 관계 «extend»

조건에 따라 선택적 실행. 화살표: 확장 UC → 기본 UC

8. 모듈 설계 & 인터페이스

EAI 4유형 / 미들웨어 / REST vs SOAP

EAI (Enterprise Application Integration) — 기업 앱 통합 방식 4가지
포인트-투-포인트

앱 간 1:1 직접 연결. 단순하지만 연결 증가 시 관리 복잡(n²).

허브 앤 스포크

중앙 허브가 모든 연결 관리. 허브 장애 시 전체 중단.

메시지 버스

공유 통신 채널에 발행·구독. 느슨한 결합, 확장성 좋음.

하이브리드

허브앤스포크 + 메시지 버스 혼합. 대규모 기업 유연 활용.

인터페이스 구현 방식
REST API
  • HTTP 메서드(GET·POST·PUT·PATCH·DELETE)
  • 자원(URI) 기반, 무상태(Stateless)
  • JSON 데이터 교환, 경량·범용
SOAP
  • XML 기반 메시지, 엄격한 규격
  • WSDL로 서비스 명세
  • WS-Security, 금융·공공기관
미들웨어 유형
  • DB 미들웨어: ODBC, JDBC
  • MOM: 메시지 지향 미들웨어 (비동기)
  • TP-Monitor: 트랜잭션 처리
  • WAS: 웹 애플리케이션 서버
  • ORB: CORBA 기반 객체 요청
